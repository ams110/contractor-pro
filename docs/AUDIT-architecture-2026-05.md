## 2. المعمارية والامتثال القانوني

### ❌ 2.1 التوقيع الرقمي

**أ) غير مفروض على كل الإدخالات:**

| العملية | توقيع مطلوب؟ | المرجع |
|---------|:---:|--------|
| إضافة مشروع | ✅ | `ProjectsScreen.jsx:94` |
| حذف مشروع/قبضة/مصروف/دفعة | ✅ | عدة أماكن |
| إضافة دفعة راتب | ✅ | `FinanceScreen.jsx:690` |
| **إضافة قبضة (مدخول)** | ❌ | `AddReceiptSheet.jsx:95` |
| **إضافة مصروف** | ❌ | `AddExpenseSheet` / `ProjectFinanceTab` |

**ب) ليس توقيعاً قانونياً حقيقياً:**
- تحدّي WebAuthn يُولَّد في المتصفح ولا يُتحقَّق منه على السيرفر (`useBiometricConfirm.js:48-58`) → بوابة محلية فقط.
- `signature_log` يُحفظ fire-and-forget ومنفصل عن السجل، بدون hash للمحتوى (`useBiometricConfirm.js:24-32`) → لا يكشف التلاعب.
- PIN = SHA-256(pin+email) محلي، 4 خانات قابلة للكسر فوراً.

**الخيارات (قرار مؤجَّل — يحتاج استشارة محاسب/محامٍ):**
- **(أ) تأكيد + سجل تدقيق قوي:** فرض البصمة/PIN على كل إضافة + ربط السجل بالـ id والمبلغ. شغل صغير، أثر تدقيق داخلي.
- **(ب) توقيع رقمي قانوني:** توقيع hash السجل + تحقق على السيرفر (Edge Function) + قفل التعديل عبر RLS. شغل كبير، قيمة إثباتية (`חתימה מאובטחת`).

### ❌ 2.2 الإيصال (קבלה)
- الترقيم التسلسلي التلقائي **RCP-XXXX غير مطبّق إطلاقاً** رغم أن الواجهة تَعِد به (`AddReceiptSheet.jsx:256`).
- `ref_number` يخزّن الرقم الخارجي/الشيك اليدوي، وقد يكون `null` (`AddReceiptSheet.jsx:110`).
- صورة الإيصال اختيارية → يمكن تسجيل قبضة بلا קבלה وبلا رقم. **مخالفة قانونية.**
- **الإصلاح:** توليد رقم تسلسلي تلقائي مضمون التفرّد (يفضَّل عبر تسلسل/RPC في قاعدة البيانات لا في العميل) + جعل الإيصال إلزامياً.

### ⚠️ 2.3 ربط المشروع بالمصلحة — **القرار: مصلحة واحدة لكل مشروع**
- يوجد نموذجان متوازيان: عمود `projects.business_id` (1:1) وجدول `project_businesses` (n:n).
- **القرار المعتمد:** مصلحة واحدة فقط لكل مشروع → الاعتماد على `business_id` و**إزالة** جدول `project_businesses` وهوك `useProjectBusinessLinks` لاحقاً.

### ⚠️ 2.4 ازدواج مسارات الإدخال
- ثلاث مسارات لإدارة مالية المشروع بكود مكرّر: `ProjectFinanceTab.jsx` (به نسخة داخلية من sheets)، `components/sheets/AddReceiptSheet.jsx` (نسخة موحّدة)، و`ProjectsScreen.jsx`.
- **الإصلاح:** توحيدها بمسار/مكوّن واحد بسلوك متّسق.

## 3. القرارات المعتمدة من المالك
1. **أساس التكلفة:** المدفوع (cash) — لتوحيد الأرقام وحل الازدواجية.
2. **المشروع↔المصلحة:** مصلحة واحدة فقط لكل مشروع.
3. **التوقيع:** القرار بين (أ) و(ب) مؤجَّل لحين استشارة محاسب/محامٍ.
4. **النطاق الحالي:** توثيق فقط — لا تعديلات كود بعد.
